# 경비관리 AI 서비스 개발안

## 🎯 1. 서비스 핵심 목표 (Core Objectives) - 멀티테넌트 기반

* **통합 정산 파이프라인**: 다수의 회사와 산하 사업장들이 하나의 시스템 안에서 각자의 독립적인 영수증 정산 프로세스, 사칙 챗봇, 컴플라이언스 검증을 수행할 수 있는 완벽한 멀티테넌트 SaaS/내부망 솔루션을 구축합니다.


* **공통 도메인 기반의 지능화**: 모든 기능(1, 2, 3)이 테넌트별로 파편화되지 않고, 동일한 핵심 도메인(용도, 사칙, 영수증 내역)을 참조하여 유기적으로 동작하게 함으로써 AI의 판단(환각 방지)의 일관성을 보장합니다.


* **최종 목표**: 영수증 입력부터 규정 검토, 환각 수정, 최종 결의서 파일 생성까지 전 과정을 자동화합니다.

## 🏗️ 2. 멀티테넌트 아키텍처 및 공통 도메인 설계 (Architecture & Domain Design)

### 핵심 도메인 정보의 구조화

시스템의 모든 AI 에이전트와 도구는 각 테넌트의 ID(company_id, workplace_id)를 기준으로 아래의 공통 도메인에 접근합니다.

* **용도 도메인 (정형 데이터)**: DB(PostgreSQL/Oracle) 내 company_receipt_rules 등 테넌트별 우선순위, 적용 조건(직책, 금액, 시간, 키워드)과 결과(식대, 접대비 등)를 저장합니다.


* **규정/사칙 도메인 (비정형 벡터 데이터)**: 벡터 DB(Qdrant 등) 내 인덱스에 테넌트별 사규, 마케팅 지침, 경조사 기준 등 문서 데이터를 저장합니다.


* **승인 내역 도메인 (정형/벡터 하이브리드)**: 과거 영수증 승인 이력을 RDB에 정형 데이터로 저장하고, 유사 사례 매칭용으로 벡터 DB에 임베딩하여 저장합니다.



### 시스템 기술 스택

* **오케스트레이션 (환각 제어 및 워크플로우)**: LangChain, LangGraph


* **RAG 및 비정형 검색 도구**: LlamaIndex + Qdrant (또는 밀버스 등 벡터 DB)


* **관계형 데이터 및 규칙 엔진**: PostgreSQL / Oracle


* **API 백엔드**: FastAPI (모든 요청에 company_id, workplace_id를 헤더/토큰으로 전달받아 멀티테넌트 컨텍스트 유지)

## ⚙️ 3. 기능별 세부 구현 및 도구 연계 방향

### 기능 1: 영수증 사용 용도 자동 추천 (Rule & History 기반)

* **작동 방식**: 영수증 데이터와 사용자의 테넌트 정보(company_id, workplace_id)를 입력받습니다.


* **사용 도구 (LangChain Tools)**: Rule_Engine_Tool을 통해 DB에 질의하여 테넌트(사업장 우선, 공통 차선)의 정형화된 '용도 도메인' 규칙과 매칭합니다.


* **History_Lookup_Tool**: 규칙 매칭 실패 시, 해당 테넌트의 과거 '승인 내역 도메인'을 검색(Few-shot)하여 유사한 사례를 바탕으로 용도를 추론합니다.



### 기능 2: 사내 규정 문서 챗봇 (RAG)

* **작동 방식**: 사용자의 질문을 받아, 해당 사용자가 속한 테넌트의 사칙 문서에서만 답변을 찾습니다.


* **사용 도구**: LlamaIndex 기반 RAG 시스템.


* **핵심 설계**: Qdrant 같은 벡터 DB에 문서를 저장할 때 메타데이터(Metadata)로 company_id와 workplace_id를 반드시 부여합니다. 검색 시 이를 필터 조건(Payload Filtering)으로 사용하여 타 회사의 사칙이 유출되거나 섞이는 것을 방지합니다 ('규정/사칙 도메인' 참조).



### 기능 3: 영수증 컴플라이언스(규정 준수) 자동 확인 및 루프

* **작동 방식**: 법인카드의 승인정보와 사용자를 확인 후 실제 테넌트의 '사칙'에 위배되지 않는지 검증합니다.


* **사용 도구 및 LangGraph 흐름**: 법인카드의 승인정보와 사용자 정보를 받습니다. Compliance_Check_Tool(기능 2의 RAG 엔진을 활용)을 호출하여 '규정/사칙 도메인'에 질의합니다.  (예: "우리 회사 본사(workplace_id) 규정상, 대리가 야근식대로 15,000원을 쓰는 것이 합당한가?")


* **검증 루프**: 위반 판정 시 LangGraph의 엣지(Edge)를 통해 기능 1 노드로 반려 사유(예: "식대 한도 초과")와 함께 되돌려 보내 재추천을 요구합니다.



### 최종: 정산서 / 결의서 자동 생성

* **데이터 수집**: 기능 3의 환각 검증 및 컴플라이언스 확인 루프를 무사히 통과한 데이터들만 LangGraph의 최종 State에 모입니다.


* **파일 생성**: 해당 테넌트(company_id) 전용의 Excel 또는 PDF 템플릿 양식을 불러와 파이썬 라이브러리(openpyxl 등)를 통해 데이터를 매핑하고 완성된 파일을 생성합니다.

## 🚀 4. 개발을 위한 추가 조언 (Next Steps)

* **벡터 DB 멀티테넌시 설계**: LlamaIndex와 Qdrant를 연동할 때, 회사별로 인덱스(컬렉션)를 아예 분리할 것인지, 아니면 하나의 큰 인덱스에 메타데이터 필터링을 걸 것인지 결정해야 합니다. 확장이 잦다면 메타데이터 필터링 방식이 유지보수에 유리합니다.


* **공통 도메인의 동기화 로직**: 관리자가 웹에서 '용도' 규칙을 변경하거나 '사칙' 문서를 업데이트했을 때, 즉시 시스템(DB 및 벡터 DB 인덱스)에 반영되는 파이프라인(Data Ingestion Pipeline) 구축이 RAG 성능에 매우 중요합니다.
